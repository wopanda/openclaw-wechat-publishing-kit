# 图片替换协议

当 `material-to-graphic-report` 已经产出飞书图文原型，后续又拿到了真实图片或生图结果时，按本协议把图片从“占位状态”推进到“正式成品状态”。

## 目标

确保图片替换不是手工乱贴，而是可追踪、可回填、可复用、可继续修改的标准流程。

## 第一性原则

1. 文档中的图片位不是注释，而是未来真实图片的固定坑位。
2. 替换动作应以 `slot_id` 为锚点，而不是靠肉眼找“图1”“图2”。
3. 替换图片时，优先保留上下文文字、段落位置和图注节奏，只替换图片本体。
4. 如果图片还不够好，允许保留图片位继续迭代，不要为了“看起来完成”而硬替。

## slot_id 规则

每张图都必须有稳定的 `slot_id`，推荐格式：
- `cover_01`
- `sec2_compare_01`
- `sec3_structure_01`
- `sec4_evidence_01`
- `sec5_close_01`

规则：
1. 同一文档内不得重复。
2. 一旦进入飞书文档原型，不要中途随意改名。
3. 后续的图片文件、提示词记录、替换日志，都围绕同一个 `slot_id`。

## 图片位状态机

每个图片位只有以下几种状态：

- `待生成`：只有占位与提示词，还没有真实图片
- `已生成待替换`：图片已拿到，但还没写入飞书文档
- `已替换`：真实图片已进入文档
- `待重做`：图片效果不满意，需要重新生成

不要发明更多状态，保持简单。

## 替换输入

替换动作至少需要：
- `doc_id` 或文档链接
- `slot_id`
- 对应图片文件或图片 URL
- 原图注（如要保留）
- 是否保留原图片位文本说明

## 标准替换流程

### 第一步：定位图片位
根据 `slot_id` 找到文档中的目标图片位块。

### 第二步：确认当前状态
检查该图片位是：
- 待生成
- 已生成待替换
- 待重做

如果已经是 `已替换`，先确认是否为覆盖更新，而不是重复插图。

### 第三步：插入真实图片
如果有可公开访问的图片 URL：
- 优先用 `<image url="..."/>` 方式写入文档

如果只有本地图片：
- 通过文档媒体工具追加或插入图片

### 第四步：保留或更新图注
默认保留原图注，除非新图片语义发生明显变化。

### 第五步：清理占位信息
替换后，删除以下占位字段：
- 状态
- 画面描述
- 生图提示词
- 风格
- 比例

通常保留：
- 图片标题（可选）
- 图注

### 第六步：更新状态记录
把该 `slot_id` 的状态改为 `已替换`，必要时在交付层更新替换进度。

## 飞书文档中的推荐替换方式

### 推荐方式 A：保留图片标题 + 图注
适合正式成品。

替换前：
```md
### 图片位｜sec2_compare_01｜旧流程 vs 新流程
- 状态：待生成
- 用途：展示旧流程与新流程差异
- ...
- 图注：差别不在信息量，而在是否还要自己先跑那一轮重复动作
```

替换后：
```md
### 旧流程 vs 新流程
<image url="https://..." width="960" height="720" align="center" caption="差别不在信息量，而在是否还要自己先跑那一轮重复动作"/>
```

### 推荐方式 B：只保留图片 + 图注
适合更轻成品风格。

## 不推荐的替换方式

1. 直接把新图片追加到文档末尾，不处理原图片位。
2. 贴了图片但原占位说明全部还留着，导致文档像施工现场。
3. 不按 `slot_id` 替换，而是凭感觉找位置。
4. 一张图替换多处不同用途的图片位。

## 多轮生图时的处理

如果同一个 `slot_id` 生成了多版图片：
- 保留最新采用版
- 旧版不要混进正文
- 可在外部记录中标记 `v1 / v2 / final`
- 进入正式飞书文档时，正文只保留当前采用版

## 失败与降级策略

如果图片生成失败或暂时不可用：
1. 保留原图片位
2. 状态改成 `待重做`
3. 不要为了“完整”而随便塞一张不匹配图片

如果文档工具暂时只能追加图片、不能精准替换：
1. 先保留原文档作为原型
2. 另做一版“成品稿”文档
3. 在成品稿里直接用真实图片重新装配

## 输出检查清单

替换完成前，至少检查：
1. 是否每张图都按 `slot_id` 对应到正确坑位
2. 是否删除了占位状态和提示词残留
3. 图注是否仍然成立
4. 文档节奏是否比替换前更像成品，而不是更乱
5. 交付层是否更新了替换进度
