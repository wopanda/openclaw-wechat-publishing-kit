# 飞书落地策略

当目标不是普通 Markdown，而是“在飞书里点开就能预览”的图文报告时，按下面的策略执行。

## 目标

最终交付不是一段聊天回复，而是一个飞书内可阅读的图文文档原型。
当前没有生图结果时，也要让用户在飞书里看到：
- 结构完整
- 图位清楚
- 每张图要表达什么已经写清
- 后续可以按 slot 替换成正式图片

## 默认交付顺序

### 路线A：文档优先（推荐）
适用于：
- 内容超过一屏
- 有 2 张以上图片位
- 用户需要预览整体成品感
- 后续还要补图、改图、替换图

执行：
1. 先生成飞书图文稿 Markdown
2. 创建飞书文档原型
3. 把结构化图片位直接写进文档
4. 返回文档链接

### 路线B：消息优先
适用于：
- 只是快速确认方向
- 内容很短
- 当前只想先看标题、提纲、图位清单

执行：
1. 在消息里发简版
2. 一旦用户确认继续，立即转成飞书文档原型

## 飞书文档内的呈现规则

### 1. 把图片位当成“内容块”
不要把图片位写成技术注释。
应把它们写成用户也能看懂的预留块，至少包含：
- 图片名称
- 用途
- 图注
- 当前状态（待生成）

### 2. 先有成品感，再等图片
即使没有正式图片，文档也要看起来像一篇完整图文稿。
所以图片位出现时，前后要有：
- 标题
- 导语
- 正文段落
- 图位块
- 图注
- 下一段内容

### 3. 一张图一个 slot_id
后续替换时，按 slot_id 逐位回填。
不要使用模糊命名，如“图1”“图2”。
推荐：
- `cover_01`
- `sec2_compare_01`
- `sec3_structure_01`
- `sec4_close_01`

## 后续补图与替换策略

### 当前阶段
- 生成结构化图片位
- 明确每张图的提示词、比例、风格、图注
- 文档中保留 slot_id

### 生图能力接入后
1. 按 slot_id 逐位生成图片
2. 将图片插入飞书文档
3. 用真实图片替换原图片位占位块
4. 保留图注和上下文段落不变

## 什么时候必须建飞书文档

出现以下任一情况时，优先建飞书文档而不是只回消息：
- 用户明确要“发我链接预览”
- 内容目标就是图文报告
- 需要看整体排版节奏
- 后续还要继续补图或迭代版本
- 需要形成可反复修改的工作底稿

## 当前环境的现实约束

如果飞书文档创建工具因权限问题不可用：
1. 继续把图文稿 Markdown 准备好
2. 保留完整图片位
3. 明确告诉用户：当前卡在平台权限，不是内容结构没准备好
4. 权限一恢复，就直接把现成预览稿推成飞书文档

## 与旧经验的衔接

当前本地可回查的相近思路包括：
- 第5章内容生产线：已明确“选题 → 调研 → 提纲 → 初稿 → 配图建议 → 终审”的链路
- 旧 chapter_12 / chapter_13：已出现“配图建议”层

这意味着：
我们现在不是从零开始，而是在把旧的“配图建议”升级成：
- 结构化图片位
- 可后续生图替换的 slot 体系
- 飞书文档原型落地
