# 端到端执行清单

这个清单把 `material-to-graphic-report` 从“拿到素材”推进到“飞书图文成品”拆成固定动作。

目标：
- 不靠临场发挥
- 不漏步骤
- 可分阶段推进
- 可随时停在一个稳定中间态

## 阶段 0：收需求

### 0.1 明确输入
至少确认：
- materials
- goal
- my_points
- output_style
- deliver_to_feishu

### 0.2 明确当前模式
三选一：
- `纯原型模式`：先做图文预览，不生图
- `半闭环模式`：做图文预览 + 生图输入准备
- `闭环模式`：做图文预览 + 生图 + 图片替换

### 0.3 明确约束
至少确认：
- 是否要求中文直接进图
- 是否需要飞书内部预览
- 是否需要后续替换成正式成品

---

## 阶段 1：内容重组

### 1.1 拆解素材
输出：
- 可复用结构
- 可复用证据
- 不应直接沿用的部分

### 1.2 注入用户观点
输出：
- 新角度
- 新标题候选
- 新提纲

### 1.3 形成正文主骨架
输出：
- 飞书图文稿的正文结构

验收：
- 只看文字，也能读懂逻辑主线

---

## 阶段 2：视觉脚本

### 2.1 判断哪些地方需要图
逐段判断：
- 封面图
- 对比图
- 结构图
- 流程图
- 证据图
- 案例场景图
- 收口图

### 2.2 给每张图分配 `slot_id`
要求：
- 唯一
- 稳定
- 后续不随意改名

### 2.3 写结构化图片位
每张图至少有：
- slot_id
- purpose
- visual_type
- scene_description
- prompt
- aspect_ratio
- style
- caption

验收：
- 不生图时，也能一眼看懂每张图该做什么

---

## 阶段 3：飞书原型

### 3.1 先生成飞书图文 Markdown
要求：
- 遵循 `feishu-visual-layout.md`
- 图片位直接写进文档节奏中
- 不把图片位写成技术注释

### 3.2 创建飞书预览文档
输出：
- doc_url

### 3.3 验收原型
检查：
- 在飞书里是否能直接点开看
- 文档节奏是否像图文稿，不像施工文档
- 没有图时，占位是否仍然有成品感

验收：
- 即使还没生图，用户也能看结构与成品方向

---

## 阶段 4：生图输入准备

### 4.1 按输入契约整理生图请求
遵循：
- `image-generation-input-contract.md`

### 4.2 按提示词设计指南整理 prompt
遵循：
- `image-prompt-design-guide.md`

### 4.3 如果用户明确要求中文直接进图
额外要求：
- prompt 中显式要求正确简体中文
- 强调清晰、自然、无锯齿、无乱码
- 把中文标题 / 副标题 / 图注写进 prompt 约束

验收：
- 每张图都能独立进入生图
- 返回结果能映射回 `slot_id`

---

## 阶段 5：生图执行

### 5.1 逐图生成
默认顺序：
1. cover_01
2. 关键对比图 / 结构图
3. 证据图
4. 收口图

### 5.2 记录结果
每张图记录：
- slot_id
- 版本号
- 图片路径 / URL
- 状态：已生成待替换 / 待重做

### 5.3 内部质检
检查：
- 图的主任务有没有完成
- 中文是否正确、清晰
- 风格是否统一
- 是否更像图文报告，而不是广告海报

验收：
- 不用每张都找用户确认
- 内部先筛掉明显跑偏版本

---

## 阶段 6：图片替换

### 6.1 按 `slot_id` 定位
不要凭肉眼猜位置。

### 6.2 替换真实图片
遵循：
- `image-replacement-protocol.md`

### 6.3 清理占位字段
删除：
- 状态
- 提示词
- 画面描述
- 风格 / 比例说明

保留：
- 图片标题（可选）
- 图注

### 6.4 更新交付层
标记：
- 哪些图已替换
- 哪些图待重做

验收：
- 文档从“原型”提升为“成品稿”

---

## 阶段 7：最终交付

### 7.1 默认交付项
- 飞书文档链接
- 当前完成度
- 若未完全闭环，明确卡在哪一段

### 7.2 三种完成状态
- `原型完成`：文档已建好，图位齐全，未生图
- `半闭环完成`：文档 + 生图输入齐全，图片部分已准备
- `成品完成`：文档 + 图片替换完成，可直接展示

---

## 执行原则

1. 一次只推进一个稳定阶段。
2. 没必要每张测试图都让用户拍板；先内部筛选，再在关键节点汇报。
3. 如果已经打通飞书预览链路，优先把 skill 做完整，不要频繁停在“是否继续”的口头确认上。
4. 如果用户明确要求“逐步进行”，就按阶段汇报，不要一口气做成黑箱。

## 当前最适合的阶段汇报格式

```md
- 已完成：阶段 X
- 产物：xxx
- 验收：xxx
- 当前状态：原型完成 / 半闭环完成 / 成品完成
- 下一步：阶段 X+1
```
