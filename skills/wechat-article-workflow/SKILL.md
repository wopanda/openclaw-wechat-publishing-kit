# WeChat Article Workflow

这是一个**主调度 skill**。

它不自己假装包办一切，而是负责：

1. 先读对项目规则
2. 再把输入收口成可写内容
3. 再把后链交给预览和发布 skill

---

## 一、它支持哪些输入

保留 4 个入口，不能只剩一种：

### 1）日报选点
例如：
- “展开今天 digest 第 2 个点”
- “把今天日报第 3 个点做成公众号选题”

### 2）给一篇文章
例如：
- “参考这篇文章，改成我的公众号方向”
- “吸收这篇文章的结构，但换成我的判断”

补充规则：
- 这类输入可以先用 `wechat-article` 或等价抓取能力提取干净正文。
- `wechat-article` 只负责抓参考文、拆结构、提取关键信息，不充当创作主流程的大脑。
- 外部文章只能作为参考材料，不能直接覆盖用户自己的主线、判断和口吻。

### 3）给一个主题
例如：
- “写‘数字员工为什么不是装上就能用’”

### 4）给一批材料 / 线索
例如：
- “把这些笔记整理成公众号选题”
- “根据这批材料先收口观点”

## 三、主题入口的默认取料顺序

如果用户只给了一个主题，默认按下面顺序找素材：

### 1）先找用户自己的材料
优先从：
- 主线与价值观
- 真实项目经历
- 失败教训
- 表达偏好 / 人物定位

### 2）再找用户自己的历史运行记录
优先找：
- 观点卡
- 证据包
- 历史成稿候选
- 已有样例说明

### 3）再补外部材料
例如：
- builders digest
- 用户给的参考文章
- 相关线索 / 资料 / 评论区问题

### 4）最后才补泛背景资料
泛背景只能补强，不能反过来替代用户自己的判断。

## 四、标准执行顺序

无论从哪个入口进来，都统一收口成这条主链：

### Stage 1：收口输入
判断这次输入属于：
- 日报点
- 文章改写
- 主题写作
- 材料整理

### Stage 2：产出选题卡
至少回答：
- 这次写什么
- 为什么是它
- 适合写给谁
- 最抓人的矛盾 / 变化 / 冲突是什么

### Stage 3：产出观点菜单
不要直接跳正文。
先给用户 2~4 个可选观点方向，并明确：
- 每个方向更像什么写法
- 核心主张是什么
- 结构骨架是什么
- 为什么值得选

### Stage 4：用户选观点
用户明确选了某个观点方向后，才继续往下走。

### Stage 5：调用真实经历
从个人上下文库里抽出：
- 最相关的一段真实项目经历
- 最相关的一条失败教训
- 最相关的一句长期判断

### Stage 6：起草正文
本 skill 只负责把“已选观点方向 + 真实经历 + 起草约束”打成 handoff bundle，交给当前主模型 / 当前 Agent 继续写。
正文必须是干净正文，不能混入：
- 标题候选组
- 发布导语
- 工作流说明
- 图片提示词
- 交付备注

起草时默认遵守：
- 用户自己的主线、真实经历、失败教训优先于外部样板
- 外部样板只增强表达，不定义“你是谁”
- 标题优先抓冲突、代价、误解、反常识；不要把“悬念 > 结论、情感 > 理性”写成绝对公式
- 正文要适合公众号阅读，但不要为了排版感把内容写空

### Stage 7：过质量门
按 `16-公众号内容质量门-v1.md` 判断：
- 能不能进入预览
- 是否需要返工

### Stage 7：按需进入后链
- 要飞书预览：交给 `material-to-graphic-report`
- 要草稿箱：交给 `wechat-draft-publisher`

补充规则：
- 封面图生成不前置为必经步骤。
- 是否需要图、用专属图还是兜底图，统一后置到配图状态机判断。
- 允许 `article-specific / fallback-approved / text-only / blocked-by-image` 四种状态，不把“有图”当成默认硬要求。

## 五、与后面两个 skill 的边界

### 1）material-to-graphic-report
它负责：
- 飞书图文预览
- 视觉脚本
- 图文原型装配

它不负责：
- 选题判断
- 观点拍板
- 草稿箱发布

### 2）wechat-draft-publisher
它负责：
- 微信发布前清洗
- 预检
- 推草稿箱

它不负责：
- 选题
- 写正文
- 飞书预览原型设计

### 3）本 skill 自己负责什么
本 skill 只负责：
- 读项目规则
- 做内容前链调度
- 统一收口中间产物
- 决定什么时候该交给后链

---

## 六、什么地方必须停住

遇到下面几种情况，不要硬推：

### 1）素材明显不够
写出来会空，就先停。

### 2）没有真实经历支撑
如果这篇文章强依赖用户真实经验，但上下文库里也没有，先提醒，不要假装有。

### 3）表达卡还没定住
没有核心主张、结构骨架、证据边界，不要直接跳正文。

### 4）质量门不通过
如果结果是 `rewrite-before-preview`，就先返工，不进预览。

### 5）图片或发布链报阻塞
如果状态是 `blocked-by-image` 或 `rebuild-before-publish`，显式回报，不要装作已经完成。

---

## 七、避免过度耦合的规则

### 1）项目文档是约束层，不是代码层
这个 skill 每次都要去读项目文档，
但不要把文档里的细节硬编码成不可变逻辑。

### 2）只读最小必要文件
不要默认整个 Projects 目录全量注入。

### 3）后链保持独立
预览和发布仍然是独立 skill，
不要把它们直接揉进本 skill 里。

### 4）不要内嵌外部 LLM 依赖
本 skill 不应在内部再起一套独立的外部 LLM API 调用链，
不应依赖额外的余额、SDK、单独模型配置来完成正文生成。
正文起草如需继续推进，应由当前主模型 / 当前 Agent 基于 handoff bundle 完成。

### 5）真实内容留在项目目录中更新
规则、经历、教训、样板都优先回写项目目录；
本 skill 只是每次重新读取和调度。

---

## 八、前台回报格式

每次只回 4 件事：

1. 现在做到哪一步
2. 当前结果是什么
3. 有没有发生兜底 / 降级 / 返工
4. 下一步是什么

不要把大量工程噪音直接甩给用户。

---

## 九、一句话收口

这个 skill 的稳定目标不是“自动帮用户发完一篇文章”，而是：

**先按项目规则读对上下文，再把日报 / 文章 / 主题 / 材料统一收口成可写内容，最后按需交给飞书预览和草稿箱发布。**
