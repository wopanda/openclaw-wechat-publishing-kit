# 集成边界说明

## 一句话边界

`wechat-draft-publisher` 只负责“把已经写好的内容稳定推进到微信公众号草稿箱”。

它不是：
- 选题器
- 搜集资料器
- 正文生成器
- 标题候选器
- 自动配图主系统

## 5.1 与 5.2 的分界

### 属于 5.1 的动作
- 主题拆解
- 证据池搜集
- 正文生成
- 标题候选
- 风格迁移
- 自动配图生成
- 插图 prompt 设计与图位规划
- 飞书预览稿生成

### 属于 5.2 的动作
- 输入已成型稿件
- 转成微信可接受的 HTML / 内容结构
- 处理正文图片与封面策略
- 接收上游已生成好的插图计划并合并正文插图
- 预检配置、封面、正文图片情况
- 推进到微信公众号草稿箱
- 回写 `draft_id / media_id / 状态`

## “去 AI 味 / 润色”模块应该放在哪里

适合放在本 Skill **之前**，作为发布前上游步骤：

`成稿 / 预览稿 -> 去 AI 味 / 润色 -> wechat-draft-publisher -> 草稿箱`

这样做的好处：
- 能把“写得更像人”和“发得更稳”分开维护
- 不把发布 Skill 重新做回大而全工作流
- 更便于后续替换不同的润色模块

## 推荐的工程形态

### V1：发布主链（本仓库）
- `publish_markdown.py`
- `prepare_feishu_doc_for_wechat.py`
- `feishu_record_to_draft.py`
- `check_wechat_connection.py`

### 可选上游模块
- `humanize` / `polish`
- `markdown->wechat-html style theme`
- `preview reviewer`

## 对写书最稳的口径

- 5.2 写的是“发布链 / 排版链 / 草稿箱链”。
- 可以提“发布前还可以串一个去 AI 味模块”，但不要把 5.2 写成“从选题一路自动写到发布”的整链案例。
- 如果正文要举例，优先用“现成预览稿 / Markdown 稿件如何进草稿箱”，而不是“AI 从零写一篇文”。
